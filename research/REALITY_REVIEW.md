# Reality Review — the weekly process

Reality becomes the roadmap here. This is the loop that turns raw sessions
into product truth and keeps the documentation honest.

## The weekly cycle (during First Light: twice weekly)

1. **Collect** — every OBS/INT/VERB filed since last review (same-day filing
   rule; memory decays by morning).
2. **Cluster** — group observations by journey + root cause, not by merchant.
3. **Apply the Rule of Three**
   - 1 merchant → anecdote: stays in its OBS file, untouched.
   - 2 merchants → possible pattern: opens/updates an H-## as Open with both
     citations.
   - 3 independent merchants → **product truth**: H-## moves to
     Validated/Rejected; triggers the documentation workflow below.
   - S1/S2/trust issues bypass counting entirely → immediate friction entry +
     hotfix path (playbook/FRICTION_LOG.md).
4. **Update the register** — every touched H-## gets its evidence lines and
   confidence adjusted in the same commit as the new observations.
5. **Feed the matrix** — surviving patterns become candidates in
   playbook/PRIORITIZATION.md with their evidence attached.

## Documentation-update workflow (reality overrides design)

When a product truth contradicts standing documentation:

1. Open a branch `reality/<short-name>`.
2. Update the contradicted document (Platform Bible, Product Compass entry,
   ADR *consequences* section, UX law, copy guideline) to state the new truth.
   - Frozen ADR *decisions* are not silently rewritten: add a dated
     **Reality Note** section citing the H-## and evidence; if the decision
     itself must change, that is a STOP-and-decide moment per standing law.
3. The PR description cites: H-##, the three (or S1/S2) evidence files, and
   the sentence being replaced.
4. Merge = the documentation is honest again. **Never** the reverse order —
   reality is never reinterpreted to preserve a paragraph.

## Bias checklist (run at every review)

- **Founder proximity**: was any evidence generated by us? (Probe data is
  marked and never counts toward the Rule of Three.)
- **Politeness bias**: recruited-friendly merchants over-praise; weight
  behavior over compliments — a delighted quote with an abandoned journey is
  an abandonment.
- **Solution capture**: merchants propose fixes ("add a button for…"); file
  the underlying friction, not the proposed feature.
- **Recency bias**: yesterday's vivid session does not outrank last week's
  three-merchant pattern.
- **Survivorship**: the merchant who churned is interviewed with the same
  script if at all possible; their absence is the loudest observation.

## Quality bars

- Observation quality: expectation written BEFORE the session; timestamps;
  verbatims separated from interpretation.
- Interview quality: script followed, deviations noted, silences kept.
- Evidence quality: every recommendation traces to citable files; confidence
  stated; "none — working as intended" is a valid, valuable finding.
