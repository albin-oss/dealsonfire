# DECISIONS.md — DOF Platform Implementation Decisions

Ambiguities resolved with the safest production-ready default. Binding docs: ADR-001…004, BLUEPRINT-001/002, REVIEW-001.

**D-01 — `request_idempotency_keys` table added.** BLUEPRINT-001 §4 mandates `Idempotency-Key` behavior (stored 24h, replayed response) but its §2 table list has no storage for it. Added to migration 0001: PK (key, endpoint, actor). Rows older than 24h are purged by the outbox dispatch task.

**D-02 — `audit_logs` is partitioned; `domain_events` is not (yet).** BLUEPRINT §2.15/§2.16 ask for monthly partitioning of both. PostgreSQL requires the partition key in every unique constraint; `domain_events.UNIQUE(aggregate_type, aggregate_id, sequence)` is the optimistic-concurrency guard and cannot include `occurred_at` without weakening it. Resolution: `audit_logs` (no conflicting unique) is range-partitioned by month from day one with an `audit_logs_ensure_partition()` helper + DEFAULT safety partition; `domain_events` stays unpartitioned with the constraint intact — revisit with hash-by-`business_id` partitioning when volume demands (the constraint survives hash partitioning on business_id only if included; tracked as a scale task).

**D-03 — PublishableStoreSpec's listing clause is behind a port.** The spec (BLUEPRINT §4) includes "≥1 published listing", but Catalog is Module 2 and PROMPT 003 requires publish to work end-to-end now. The spec consumes a `ListingReadinessPort`; the Module 1 adapter reports `catalogAvailable: false`, which skips the clause. Module 2 wires the real adapter — no spec change needed.

**D-04 — Identity stub.** `IdentityPort` with two adapters: `dev` reads `x-dof-user-id` (must be a UUID) and **refuses to run when NODE_ENV=production**; `session` is the future Identity-domain adapter and fails closed (401) until it lands. Selected via `NUXT_IDENTITY_MODE`.

**D-05 — `GET /workspace` for a user with no merchant account returns 200** with an empty workspace (`merchant: null`), not 404 — the client uses this to render the "start selling" opportunity (Opportunity First).

**D-06 — Explicit entitlement grants bypass the tier minimum but never the trust minimum.** Tier is commercial; trust is safety (ADR §0.3). A manual grant can give a Starter business an Established capability, but nothing skips verification.

**D-07 — Merchant eligibility at business creation** = `merchant_accounts.status <> 'deactivated'`. Platform-wide ban checks belong to the Administration domain integration (not yet built).

**D-08 — Rate limiting is per-instance in-memory** (sliding window, BLUEPRINT §9 limits). Distributed limiting moves to Vercel KV when it lands; the `RateLimiterPort` interface is already in place.

**D-09 — Integration tests run against real PostgreSQL.** Locally via `embedded-postgres` (real PG binaries, no Docker on this machine); in CI via a postgres:17 service (`TEST_DATABASE_URL`). No mocked database anywhere.

**D-10 — Audit "middleware" placement.** Accepted commands write their audit row **inside the command transaction** (atomicity beats literal middleware); the endpoint wrapper additionally audits *denied sensitive* commands (which have no transaction). Equivalent to BLUEPRINT §9 with stronger guarantees.

**D-11 — First real outbox consumer: `StandingConsequencePolicy`** (ADR §5.4 — suspended/banned business ⇒ enforcement hold on all its stores). The Administration API that emits `business.standing_changed` arrives in a later module; the policy + `event_deliveries` idempotency are integration-tested by appending the event directly.

**D-12 — Explicit h3 imports in `server/`** (no Nitro auto-imports). Keeps handlers unit-mountable in tests and consistent with the framework-free discipline of `domains/`.

**D-13 — Workspace query reads aggregate tables directly** (memberships → businesses → stores joins). BLUEPRINT §7 names an `rm_workspace_overview` projection; it becomes worthwhile with Pulse (Module 3). The query is per-user, indexed, and low-traffic — a projection now would be unjustified complexity (ADR: complexity only when justified).

**D-14 — Merchant context resolves per-endpoint, not via global middleware.** BLUEPRINT §1 names an `02.merchant-context` middleware; the implementation fuses context resolution with the triple gate in `resolveAndAuthorize` because the business/store being acted on comes from route params a global middleware cannot see. (REVIEW-001 L-8.)

**D-15 — Outbox claims are partition-serial, ordered by a DB-assigned sequence.** Only the oldest pending row of each partition is claimable, so a backing-off (or currently-locked) event holds its partition — and only its partition — preserving per-business ordering across retries and concurrent workers. "Oldest" is decided by `outbox_events.seq` (identity column, migration 0002), NOT the UUIDv7 id: UUIDv7 is not monotonic within one millisecond, and multi-event appends from a single transaction land in the same millisecond (caught by the PROMPT-005 verification run — the uuid-ordered version was probabilistically wrong). Dead-lettering (10 attempts) unblocks a poisoned partition. Additionally, known event payloads are schema-validated (contracts/schemas/events/payloads.ts) before consumers run; invalid payloads dead-letter immediately since they can never become valid. (REVIEW-001 H-1, M-6.)

**D-16 — Derived handles never fail.** Platform-generated handle suggestions fall back through numbered variants, then random `store-<suffix>` — including names that slugify to nothing (CJK, Arabic, emoji) or collide with reserved words. Only merchant-CHOSEN handles surface HANDLE_TAKEN, with suggestions. (REVIEW-001 H-3.)

**D-17 — In-flight idempotency claims are reclaimable after 60s.** A crash between command commit and response storage must not wedge the client until the 24h purge; stale NULL-status claims are atomically reclaimed on the next attempt. (REVIEW-001 M-1.)

**D-18 — Staff roles are CHECK-constrained in the database** (migration 0002). Custom roles (Established tier, future) will extend the constraint by migration — deliberate friction for a security-relevant list. (REVIEW-001 L-5.)

**D-19 — `partition_key` is the owner-defined ordering scope** (ADR-003 W3, ADR-004 rule 15). It is NOT universally `business_id`: Merchant orders business-scoped events by `business_id` and everything else by aggregate id (`orderingScopeOf` in the domain — the previously-implicit `?? aggregate_id` fallback, now a named, documented contract). Future domains define their own scope in their event-store implementation.

**D-20 — Trace is stamped at `EventStore.append`, not in aggregates** (ADR-004 C1). Aggregates state facts and stay request-context-free; commands pass `traceFromRequest(requestContext)` (correlation only — a command has no causing *event*), and event consumers pass `traceFromEvent(source)` (inherit correlation, causation = source event id). One request becomes one traceable tree without polluting the domain model. Columns are nullable: pre-0003 events and pure system schedules legitimately carry no trace.

**D-21 — Grant-level immutability requires a non-owner application role** (ADR-004 C4). `db/grants/immutable-tables.sql` (applied via `npm run db:grants` after every migration deploy — partitions are re-covered on each run) revokes UPDATE/DELETE/TRUNCATE on `audit_logs` and `domain_events` from the app role. Production MUST run the app as a dedicated role (`dof_app`-style), distinct from the migration/owner role — an owner or superuser connection bypasses grants by definition. Verified in integration by SET ROLE + expecting 42501.

**D-22 — Platform machinery is a top-level library, instantiated per domain** (BLUEPRINT-002 Batch 1, K1). Event store, outbox dispatcher, audit log, idempotency store, UnitOfWork, and the projection registry live in `platform/` — framework-free, domain-agnostic, table names and ordering-scope functions supplied at composition time. Each domain gets its OWN event/outbox/delivery/audit tables (ADR-003 §3, ADR-004 rule 15); merchant's instances are byte-identical to Module 1 behavior. `platform/` may import only `shared/` + node built-ins + pg; domains may import platform, and domain LAYERS may import only its pg-free type modules (`platform/types`, `platform/events`) — boundary-linted. Branded ids remain a domain concern: the platform envelope uses plain strings; domains narrow via re-export.

**D-23 — `request_idempotency_keys` is platform-owned** (BLUEPRINT-002 K2). The key is (key, endpoint, actor) with no domain data; one table serves every domain's endpoints. Manifest updated; no schema change.

Ambiguities resolved with the safest production-ready default, per PROMPT 003. Binding docs: ADR-001 v1.1, BLUEPRINT-001 v1.0.

**D-24 — Platform Foundation completed (IMP-PLT-001) with two documented conflicts.** Added to `platform/`: clock, structured logger (JSON + redaction), metrics port, cursor pagination, repository base, consumer registry, event replay (delivery-ledger-aware, per-aggregate order), health/readiness checks, typed config + env feature flags, semantic-free security utilities, projection versioning (table-comment stamped), configurable outbox retry strategy. **Conflict C-1:** permission/capability helpers were requested in platform but REMAIN in merchant shared-kernel — ADR-003 §3 assigns RBAC/capabilities to Merchant and F2/F7 froze those interfaces; platform holds only semantic-free helpers (tenancy masking, sanitization, redaction). **Conflict C-2:** the event schema registry stays in `contracts/` — platform may not import zod/contracts (D-22); platform owns the consumer registry and replay, which are schema-free.

**D-25 — ESLint is a CI quality gate** (flat config, typescript-eslint). Scope: correctness rules only (eqeqeq, no-var, prefer-const, unused vars, inline type-imports, preserve-caught-error); formatting is deliberately not enforced; `no-explicit-any` off (strict tsconfig governs; tests use `any` for HTTP bodies). Runs in CI between the data gate and typecheck.

**D-26 — Log redaction is token-matched, never substring-matched** (REVIEW-002 H-2). Keys split on delimiters + camelCase; a key redacts when a token EQUALS a redacted key, or — for long redacted keys (≥6 chars, e.g. `password`) — when a token contains it. Rationale: substring matching on `ip` redacted `shipping_method`, `description`, `membership_id` (empirically proven), gutting commerce observability and destroying actor traceability. The commerce-field probe fixture is the permanent regression test.

**D-27 — Replay cursors are `(aggregate_id, sequence)` keysets and replay validates payloads** (REVIEW-002 H-1/M-1). The resume key IS the sort key — the uuid-based `afterEventId` marker was removed after it was shown to silently skip whole aggregates across pages (sort order and resume order disagreed). Replay accepts the same payload-validator registry as the dispatcher: invalid events are counted and skipped, never delivered — a dead-lettered poison event cannot be resurrected by replay. Additionally (M-2 deep form): projection rebuilds normalize `__shadow`-suffixed index names after the atomic rename, or the SECOND rebuild of any index-bearing projection would collide.

**D-28 — Product Aggregate design decisions (IMP-COM-001).** (a) `commerce.variant.price_changed` is emitted alongside the brief's minimum event list — ADR-002 §13 defines it as a frozen high-fan-out fact (Orders revalidation, price history, fraud signals) and the ADR wins. (b) `restore()` lands on ACTIVE and is the ONLY archived→active door; `activate()` is draft→active only (same target, different meaning — I8/I9). (c) Option-surgery protocol: adding an axis requires an explicit per-variant value assignment (the domain never guesses); removing a value requires it be unused; removing an axis requires variants stay distinct after the collapse. (d) Zero declared options ⇒ exactly one variant (the empty combination is unique) — a theorem of I3, not a special case. (e) Media: one hero per (product | variant) scope; same asset attachable once per scope. (f) Sale windows live on the variant, must strictly undercut base price, and share its currency; scheduled transitions belong to pricing/. (g) Bounded collections (3 options / 100 variants / 50 media) are exported constants — tier-policy overrides are a future decision, not a schema change. (h) NO tables/migrations this sprint: manifest-first (ADR-004 C6) applies when persistence lands; only the `ProductRepository` contract ships. (i) Commerce payload schemas + registry-lock entries registered NOW, before any dispatcher exists — M-6 covers commerce from its first emitted event. (j) `catalog/application/` is deliberately empty this sprint: ProductFactory/ProductValidator live in `domain/` per the kernel layout (factories are domain construction rules, not use-cases); command handlers populate application/ in the persistence sprint — no placeholder files (Engineering Constitution).

**D-29 — Events describe DETECTED change, never request shape; no-ops are silent; direct mutations supersede AI provenance** (REVIEW-003 H-1/M-1, FIX-COM-001A). Platform-wide rules: (a) an aggregate computes `fields_changed` from actual before/after deltas — never from the request's keys — and emits nothing when nothing changed (an event a domain's own registered schema rejects is a self-manufactured poison pill that dead-letters and blocks its partition head); (b) `variant.price_changed` fires only on real base-price or sale changes; (c) AIProvenance entries are written ONLY by draft acceptance — any direct mutation of a field clears its provenance entry (`clearFieldProvenance`), keeping the invariant crisp: presence in AIProvenance ⇔ current content is AI-authored (ADR-001 §13.3's auditability promise). No-op edits clear nothing.

**D-30 — Product vertical slice decisions (IMP-COM-001B).** (a) Options persist as a jsonb VO document on `products`, NOT tables — the sprint brief said "option tables" but frozen BLUEPRINT-002 §2.1 decided jsonb (integrity is transactional, never relational) and the blueprint wins. (b) The repository REPLACES children on save (delete+reinsert with stable ids per BLUEPRINT §3); when Inventory adds FKs onto `product_variants`, the save strategy must become a diff — inventoried debt, not a surprise. (c) Commerce reaches merchant context ONLY through `MerchantAccessPort` (the in-monolith face of frozen F3), implemented by the composition root — the boundary lint keeps commerce out of merchant/core structurally. (d) `PATCH /products/{id}` is ATOMIC: one composite command, one transaction; a bad category rolls back a good title (Result-aware UnitOfWork). Rename/UpdateDescription/SetCategory exist as named doors over the composite. (e) Product tier limits (250/5k/50k/∞) are exported policy constants pending the policy-data system. (f) `PUT /products/{id}/media/order` added beyond the brief's 10 endpoints — the ReorderMedia command required a door. (g) Rehydration guard live: a row set failing the aggregate validator throws InfrastructureError (non-retryable) — proven by test against corruption the DB constraints cannot express (I2), while constraint-expressible corruption (dup SKUs) is blocked by the DB itself. (h) Trace helpers moved to `platform/trace.ts` (D-20 is platform law); merchant re-exports.

**D-31 — Acceptance closure decisions (IMP-COM-001C, closes ACCEPTANCE-001 C1/C2/N1–N4).** (a) Expected DB conflicts are translated to domain answers at the repository line: a 23505 on `uq_variants_business_sku` becomes `SKU_TAKEN` (409) with merchant-worded copy ("choose a different SKU, or leave it blank to auto-generate one") — never HTTP 500 for an expected business conflict. The endpoint wrapper renders any THROWN DomainError as problem+json (after releasing the idempotency claim), so infrastructure-detected conflicts and Result-returned ones read identically on the wire. (b) Option endpoints address options by NAME — the aggregate's identity for them (BLUEPRINT-002 §2.1 jsonb VO has no option ids; inventing one would be a new domain concept, which the sprint forbids). Names with spaces round-trip URL-encoded; `server/utils/params.ts` decodes and bounds them. (c) `POST /options` carries the aggregate's assignment protocol (D-28c) to the wire as `existing_variants_value` (one value for all existing variants — the Grandma path) and/or `variant_assignments` (per-variant map); a request with existing variants and neither field is refused with copy that names both shapes. (d) AddVariant on a zero-option product with an existing variant returns guidance (409) that teaches the options path before the domain's technically-correct-but-baffling "duplicate combination". (e) The default grid excludes archived (`show_archived=true` opts in; explicit `status=` always wins) and `q=` is a title ILIKE filter with escaped wildcards — a database filter, not the Search domain. (f) Money validation errors teach minor units ("send 1499 for €14.99"). (g) The media route segment is `{productMediaId}` — it names the attachment id, not the asset id. (h) The acceptance reviewer's black-box probes are now a permanent suite (`tests/integration/commerce/acceptance.test.ts`): pure-HTTP merchant journeys that attack every future change from the merchant angle by default.

Resolution: primitives wrap **Reka UI** — Nuxt UI's own headless foundation — preserving DS-1's intent (velocity + the replaceability seam) while keeping every component mountable in plain Vue for Vitest and Storybook. `@nuxt/ui` is not a dependency; the wrap boundary (`Dof*` owns every API) is unchanged. (b) The typecheck gate is now `vue-tsc` (templates type-checked; plain tsc cannot resolve `.vue`). (c) Two theme scopes shipped as CSS resolution scopes: `:root`/`[data-scope=workspace]` (light+dark re-resolution) and `[data-scope=storefront]` re-resolving the same sys roles from `--dof-brand-*` (runtime BrandKit via `brandKitStyle()`, every value falling back to the workspace resolution); status colors and ember stay platform-owned in every scope. (d) CSS is the token source of truth: `tokens.generated.ts` (names + tempo-band ms values) is generated from `tokens/*.css` and sync-checked in CI; `check:tokens` bans raw colors/px/ms/z-index and physical direction utilities outside `tokens/`. (e) The contrast defects the axe gate found on day one (info 3.47:1, dark-faint 3.22:1) were fixed at the ref-token tier — the gate architecture (fix tokens, whole system inherits) proved itself before the first feature shipped. (f) Storybook (39 stories) is the gate evidence: axe scans every story; keyboard and visual suites run against the static build; visual baselines are per-platform (darwin committed; linux baselines to be captured from a CI bootstrap run before the visual gate is enforced there). (g) R-classes are compiled into the pattern composables: `useUndoable` refuses R2/R3, `useConfirmation` refuses R0/R1 and demands a typed challenge for R3, `useProposal` requires the explanation quartet and refuses AI initiation of R3 and AI self-approval. **D-38 — OPS-001 Batch 1 decisions (OPS-001A: Locations + Ghost).** (a) Migration 0006 carries the operations quartet + `locations` only — the OPS-001A brief splits the blueprint's single migration per batch; forward-only law makes the split safe (stock tables land in 0007). (b) The Ghost Location is belt-and-suspenders idempotent: `pg_advisory_xact_lock(hashtext('ghost-location:' || business_id))` serializes the lazy ensure invoked by ListLocations (and every future stock command), while `uq_locations_default` (partial unique WHERE is_default AND active) is L1's big brother — 20-way race-tested. (c) The `commerce.product.created → ensureGhostLocation` consumer is the codebase's FIRST cross-domain consumer, registered on the commerce dispatcher with delivery-ledger idempotency; because the lazy ensure is the correctness path, the consumer is freshness-only — the safe place to pioneer the wiring (proven by replay test). (d) `MerchantAccessPort` is re-declared structurally in `operations/shared-kernel/ports.ts` and satisfied by the existing `merchantAccessAdapter` instance — no new adapter, no cross-domain import (CDC-001 §3). (e) L2's stock question is a port (`StockAtLocationPort`); Batch 1 binds `noStockRecordedYet` (a truthful statement — no stock table exists), replaced by the real `stock_items` query in Batch 2; the aggregate enforces the passed fact so the refusal path is fully unit-tested today. (f) `ops.location.write` joined the RBAC matrix (owner/manager full; staff and AI deliberately absent — CDC-001's AI-forbidden list starts here) and `ops.locations` joined the capability registry at Growth tier; the pre-R-3 `catalog.inventory` entry remains per registry law (never deleted). (g) Event payload types are TYPE aliases, not interfaces — interfaces lack the implicit index signature `NewDomainEvent<Record<string, unknown>>` assignability requires (a commerce idiom now made explicit). (h) `check:operations` seeds CDC-001 §7's contract-lock gate: sha256 snapshots of contract sources under `contracts/locks/operations/`, regenerated only via `--update` (drift fails CI), plus structural checks on migration/manifest/event-registration/OpenAPI.

**D-39 — OPS-001A hardening decisions (REVIEW-OPS-001 required fixes).** (a) **jsonb-backed VOs compare by canonical field equality, never JSON.stringify** — PostgreSQL canonicalizes jsonb key order (empirically: insertion order `line1,line2,city,region,postal,country` returns as `city,line1,line2,postal,region,country`), so serialization comparison false-positives on identical round-tripped documents and violates D-29. `addressEquals`/`operatingWindowEquals` are the pattern every future aggregate storing VO documents must follow (H-1; regression-locked at unit and HTTP level). (b) **Ghost identity is a persisted fact** (`locations.system_authored`, migration 0007 with backfill), set only by `createGhost` and CLEARED by the first merchant-authored detected change — a location someone named is a location they know about; system edits keep it invisible; the DTO's `ghost` flag is now this fact, never a heuristic (M-1). (c) **The GET-write exception is accepted and bounded:** ListLocations lazily ensures the Ghost, but the ensure short-circuits on `findDefault` BEFORE taking the advisory lock — steady-state lists take no lock and perform no write; the write occurs at most once per business lifetime and is system-authored, idempotent, and covered by the product-created consumer for the common path (M-2). (d) **Tier caps creation, never management:** `ops.locations` (Growth) gates CreateLocation only; update/close ride `ops.location.write` permission alone, so a tier downgrade can never strand a merchant unable to rename or close their own locations (M-4; downgrade-tested). (e) **Rehydration guards must patrol jsonb interiors:** kind/status corruption is already blocked by CHECK constraints (rule 23, verified — the corruption attempt itself died on `locations_kind_check`); the guard's territory is what CHECKs cannot see, so it now validates operating-window dates inside the document (the commerce I2 lesson, relocated). (f) The four REVIEW-mandated test classes (jsonb no-op regression, jsonb-interior rehydration guard, stale-save sequence-guard conflict, consumer negative path, Idempotency-Key replay) are written as Batch-2 templates in the operations integration suite.

**D-40 — Identity & Sessions decisions (WP-R1-B1, R1-B1 implementation).** (a) **Crypto library selection (T-2):** argon2id via `hash-wasm` (pure WASM — no native build, deterministic across dev/CI/prod, verified working in-sandbox) at OWASP-2024 params (m=19456 KiB, t=2, p=1, ~50ms); opaque tokens (session/recovery/guest) are sha-256-hashed at rest with constant-time compare — no reversible credential ever rests in the DB. Crypto lives in `domains/identity/infrastructure/` (boundary law: it is infrastructure like pg; the domain sees only the `PasswordHasher`/`TokenHasher` ports). (b) **The seam is filled:** `resolveAuth('session')` is now real via a Nitro middleware that resolves the httpOnly/Secure/SameSite=Lax session cookie → `event.context.auth`; the command wrapper reads it first (dev-header path untouched for local/tests). Sessions: rolling 30d / absolute 90d, token rotates at login and step-up; step-up window ≤5 min (the frozen D-04 window). (c) **Enumeration-proofing is structural:** login performs an argon2 verify against a dummy hash when the email is unknown (constant work), and unknown-email vs wrong-password return byte-identical problems; recovery-request always answers uniformly. Verified by an integration assertion comparing the two answers. (d) **A new PUBLIC endpoint wrapper** (`define-public-endpoint.ts`) mirrors the command wrapper's discipline (zod, RFC 9457, IP-keyed rate limits, correlation) minus the auth gate — the auth surface must work before a session exists. (e) **WebAuthn** ships real (register+authenticate ceremonies via `@simplewebauthn/server`, password always the fallback — never passkey-only) with an injectable `ChallengeStore`; the in-memory adapter is single-instance and multi-instance challenge persistence is recorded debt (WP §8 top risk; password path unaffected). (f) **Email transport** (F-XC-1 first slice) is a thin `EmailPort` + sandbox provider (test law); order emails reuse it in R1-B5 — the Notification *domain* remains R4. (g) **Client identity mode** is exposed via public runtimeConfig so the route guard enforces sessions only in session mode (dev leaves the browser open). (h) A pre-existing latent issue surfaced: the shared `MemoryRateLimiter` accumulates across integration tests (one source IP) — added a test-only `reset()` called in beforeEach; not a production concern (real traffic has many IPs and real windows).

**D-41 — Identity Vertical Slice 1 decisions (R1-B1-VERTICAL-001).** (a) **CSRF is defense-in-depth over SameSite=Lax:** the session cookie is httpOnly+Secure+SameSite=Lax (the primary CSRF defense); `server/middleware/02.csrf.ts` adds an Origin/Referer same-site assertion on mutating API requests — but ONLY in session mode, because dev-header auth carries no ambient credential a foreign site could ride (which also keeps the dev-mode integration harness exempt and unbroken). Missing Origin AND Referer on a mutating request is refused (anomalous for a browser). (b) **remember-me toggles cookie persistence, not session lifetime:** `remember:true` (the default) sets the absolute-lifetime cookie; `remember:false` omits maxAge so the cookie dies with the browser session — the server-side session TTLs (30d/90d) are unchanged either way. Added additively to `loginRequest` (contract lock regenerated). (c) The verify-email page (`/verify`) confirms on mount and follows Errors-Educate: an expired/used link shows "keep using DOF in the meantime — request a fresh link," never a dead end, because verification never blocks selling (frozen §9). (d) The slice added no new endpoints — the P0–P4 surface was already complete; VERTICAL-001 delivered the missing frontend page, the CSRF middleware, remember-me, and their tests.
